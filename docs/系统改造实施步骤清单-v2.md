# 系统改造实施步骤清单 v2.0
### CreatOK 式爆款雷达 × Higgsfield 式运镜精修 —— 基于现有海外视频本地化工作台

**代码基线：** GitHub `main` @ UI **v182**（`海外视频本地化MVP/` + `overseas-loc-mvp/`）  
**文档类型：** 可执行改造步骤（含验收标准、文件锚点、不可破坏约束）  
**适用对象：** 直接认领给对应工程师；内网部署仍以 `git pull` + `验证GitHub部署包.cmd` 为准

---

## 一、升级目标（双北极星）

| 对标产品 | 我们要补的能力 | 在现有系统中的落点 |
|----------|----------------|-------------------|
| **CreatOK 式爆款雷达** | 按品类/产品自动筛「值得跟」的对标，给出选题理由与结构标签，降低运营选片成本 | 已有 `tiktok_collector/`、`videos_meta.csv`、`video_analysis.csv`、爆款卡片网格；**缺综合雷达分、趋势解释、选题看板** |
| **Higgsfield 式运镜精修** | 每镜可选运镜类型（推/拉/摇/静态等），生成前可确认构图，成片更可控 | 已有 `storyboard[]`、`seedance_prompt`、白底主图 I2V；**缺结构化 `camera_motion`、分镜关键帧闸门、运镜 UI** |

**总原则：增量扩展，不推翻现有五步主链路。**

```
素材库 / 雷达选题 → 产品标签 → 对标 → 脚本预览 → 确认出片 → SeedDance → ffmpeg 合成 → 下载/成稿/反馈
```

## 二、现有系统锚点（改造禁区 — 逻辑不可错乱）

以下行为已在 **v175–v182** 稳定，任何批次改造**不得回退**：

| 锚点 | 位置 | 说明 |
|------|------|------|
| 爆款一键流水线 | `web/app.js` → `runViralBenchmarkPipeline()` | 拆解→脚本→排队→出片；须保留 `beginAutomatedPipelineUi()`，禁止脚本浮层与 Dock 双开 |
| 普通「开始创作」 | `runStartCreate()` | 脚本未就绪时开浮层；与爆款流水线行为区分，勿合并为同一无确认路径 |
| 成片/脚本配额解耦 | `syncDockRunButtonsDisabled()` | 成片配额用尽仍允许生成脚本 |
| SeedDance 默认增量 | `#seedanceForceRegen` 可见开关，默认 off | 勿恢复 hidden + 默认 force |
| 子进程出片 | `海外视频本地化MVP/app/seedance_bridge.py` → `overseas-loc-mvp` | `_seedance_generate_all` / `_maybe_assemble_final_video` 在 **`overseas-loc-mvp/app/main.py`**，不是工作台 `main.py` |
| 白底主图垫图 | `product_assets.py` + `output_standards` | 场景图仅 Prompt，禁止作 I2V 垫图 |
| 反馈闭环消费 | `app/feedback_loop.py` | `build_feedback_constraints` / `preview_constraints` 已可用；改数据源即可，勿重写注入逻辑 |
| 路径联接 | `ensure_legacy_paths.py` | 启动时自动 junction；新数据目录走 `01_*` / `04_*` / `05_*` |
| 内网部署校验 | `scripts/verify_deploy_repo.py` | 新模块上线须纳入校验清单 |

## 第零批（P0，爆款雷达 — CreatOK 式，建议与第一批并行）

> **目标：** 让运营「先选题、再一键跟拍」，而不是在素材库里盲选。复用现有采集与拆解数据，**不新建第二套素材库**。

### 任务 0.1：雷达评分与排序 API

- [ ] 新增 `海外视频本地化MVP/app/radar.py`（或扩展 `material_scope.py`），实现 `score_material(row, analysis) -> radar_score`
- [ ] 评分因子（可配置 `config/radar-weights.json`）：
  - 播放量 `view_count`（已有）
  - 互动率 `(like+comment+share)/view`（元数据已有字段）
  - 拆解质量：`video_analysis` 中 `hook_3s` / `video_structure` 完整度
  - 品类匹配：与 `currentProductId()` / `content_line` 一致加分
- [ ] 新增 API：`GET /api/radar/feed?product_id=&limit=24`，返回 `items[]` 含 `radar_score`、`radar_tags[]`（如「高播放」「强钩子」「结构清晰」）、`why_pick` 一句话
- [ ] **不修改** `load_materials()` 默认排序逻辑；雷达为**增量接口**，素材库原样可用

**验收标准：** 同一产品下雷达 Top10 与纯 `view_count` 排序至少有 3 条差异，且每条带 `why_pick`  
**负责角色：** 后端 + 数据　**预估：** 3–4 人日

---

### 任务 0.2：爆款雷达 UI（工作台内嵌，非新站点）

- [ ] 在 `viewImitate` / `viewWorkspace` 的精选区**上方**增加「爆款雷达」条带（或 `module-studio` 新 Tab `radar`），**复用** `feature-grid` / `viral-video-card` 样式
- [ ] `web/app.js`：新增 `loadRadarFeed()` 调 `/api/radar/feed`，卡片展示：缩略图、雷达分、标签、`why_pick`
- [ ] 卡片点击：**仍调用** `runViralBenchmarkPipeline(linkId)`（已有），仅多展示选题理由，不改变流水线步骤
- [ ] 无产品/标签时：引导 `openProductFloatPanel()`（与 v182 反推引导一致），禁止空产品请求雷达

**验收标准：** 运营可在 30 秒内完成「看雷达推荐 → 点卡片 → 进入 Dock 流水线」，且不与「开始创作」路径冲突  
**负责角色：** 前端　**预估：** 3–5 人日

## 第一批（P0，运镜精修 — Higgsfield 式）

> **目标：** 运镜从「Prompt 里碰运气」变为「结构化参数 + 可编辑 UI」。与雷达批可并行，但 **1.3 关键帧闸门** 依赖 **1.1 运镜字段** 联调。

### 任务 1.1：运镜参数化（Higgsfield 核心）

- [ ] 在 `storyboard[]` 每镜新增可选字段（向后兼容，旧脚本无此字段时默认 `static`）：

```json
"camera_motion": {
  "type": "dolly_in",
  "start_frame_focus": "人物中景",
  "end_frame_focus": "产品特写",
  "duration_sec": 3
}
```

- [ ] 运镜词表固定：`dolly_in` | `dolly_out` | `pan_left` | `pan_right` | `static` | `arc` | `crash_zoom`
- [ ] 修改产出路径（**均需保持旧 pack 可读**）：
  - `app/scene_script.py` → `build_storyboard()` 默认 `camera_motion`
  - `app/llm_script.py` → LLM JSON schema 与 prompt 增加运镜输出要求（按 `role`：钩子/痛点/方案/证明/CTA）
  - `app/script_gen.py` → 保存/合并时保留 `camera_motion`
- [ ] **`overseas-loc-mvp/app/ai_video.py`** → `sanitize_seedance_prompt()`：将 `camera_motion.type` 映射为英文运镜前缀（如 `dolly_in` → `slow dolly in,`），**追加在**现有 `seedance_prompt` 前，不删除场景/产品约束句
- [ ] `web/app.js` 脚本浮层：每镜增加运镜下拉（与「构图」「空镜」并列），写入 `data-shot-field="camera_motion.type"` 等

**验收标准：** 5 条成片盲测，运镜声明与实际一致率 ≥ 80%；旧 `ref-*` 项目无 `camera_motion` 仍可出片  
**负责角色：** 算法 + 视频 + 前端　**预估：** 4–6 人日

---

**验收标准：** 增强开/关 A/B 可感知；增强服务挂掉时仍能下载 zip  
**负责角色：** 视频生成　**预估：** 3–5 人日

### 任务 1.3：分镜关键帧确认闸门（Higgsfield 式「先构图、再动起来」）

- [ ] **子进程**：`overseas-loc-mvp/app/main.py` 拆分 `_seedance_generate_all` 为两阶段：
  1. `generate_hero_frames(slug)` — 每镜静态关键帧（可用现有 I2V 模型的首帧能力或图生图 API）
  2. `generate_shot_videos(slug)` — 仅当 `hero_frames_confirmed=true`
- [ ] 项目状态写入 `runs/ref-*/meta/pipeline-state.json`：`hero_frames` / `confirmed_at`
- [ ] **工作台 API**（`海外视频本地化MVP/app/main.py`）：代理查询/确认接口，如 `GET/POST /api/delivery/{slug}/hero-frames`
- [ ] **前端**（`web/index.html` + `app.js`）：
  - 新增「分镜关键帧确认」面板（Dock 预览区或脚本浮层内 Tab），每镜：缩略图 + 运镜摘要（如「推近 · 产品特写 · 3s」）
  - 单镜「重生成关键帧」仅调 hero API，**不触发**整片 SeedDance
  - `scriptFloatProduceBtn` / 爆款流水线出片前：若 `HERO_FRAME_GATE=1` 且未确认，则阻塞并滚动到确认面板
- [ ] **爆款流水线**（`runViralBenchmarkPipeline`）：在 `runProduceVideo` 前插入「等待关键帧确认」；可用 `state.awaitingHeroConfirm` 与现有 `videoQueue` 并存

**验收标准：** 构图问题在关键帧阶段发现；确认前不消耗 SeedDance 动态生成配额（或仅消耗关键帧额度）  
**负责角色：** 前端 + 视频联合　**预估：** 8–12 人日

**联调依赖：** 1.1 完成后再做运镜摘要展示；1.3 可用 feature flag `HERO_FRAME_GATE` 默认 off 灰度上线
